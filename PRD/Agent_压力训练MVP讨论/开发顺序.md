# 压力测试引擎 — 开发顺序

> 依据本目录下模块 PRD 与 Prompt 文档整理（角色：严肃产品合伙人；标准档 3 轮 × 3 问；拆解 + 链式追问 + 盲区快照）。  
> 与整站原始 PRD 冲突时，**以本模块文档为准**。

**拆边界、不改行为的前置重构**（压力测试壳目录、`LabPanel`/`LabContext` 轻量拆分等）见仓库根目录 [`docs/轻量重构清单-拆边界不动产品.md`](../../docs/轻量重构清单-拆边界不动产品.md)，可与本开发顺序穿插进行。

---

## 原则

1. **先跑通一条完整会话**（拆解 → 9 问含质检 → 增量更新 → 盲区累积 → 快照与判定），再扩展历史、评测、版本管理。
2. **Prompt 与业务逻辑分离**：模板常量 + 占位符注入，便于迭代与版本记录。
3. **关键规则用代码兜底**：如 `getVerdict` 与 JSON 校验，避免仅依赖模型输出。

---

## 阶段 0：对齐与契约

| 顺序 | 任务 | 说明 |
|------|------|------|
| 0.1 | 冻结模块口径 | 以本目录 Markdown 为唯一来源；拆解默认对用户不可见至会话结束（与 PRD 一致）。 |
| 0.2 | 数据契约落地 | 定义并实现类型/接口：`PressureSession`、`IdeaDeconstruction`（9 维）、`QuestionRound`、`QuestionItem`、`BlindSpotMarker`、`BlindSpotReport`、`SessionStatus`（idle → deconstructing → questioning → round_transition → completed）。 |
| 0.3 | Prompt 资产化 | 系统角色、9 条链式追问指令、初始拆解、增量更新、盲区快照 — 抽为可版本管理的模板文件 + 占位符替换层。 |

---

## 阶段 1：存储与会话骨架（MVP）

| 顺序 | 任务 | 说明 |
|------|------|------|
| 1.1 | 持久化方案 | 与现有工作台策略一致（如 localStorage / IndexedDB / 项目既有 data layer）。 |
| 1.2 | 历史列表模型 | `SessionHistoryItem`：名称、idea 预览、状态、时间、进度、完成后盲区数量等。 |
| 1.3 | 会话 CRUD | 新建、保存、列表、进入续做、删除。 |

---

## 阶段 2：拆解管线（阻塞后续追问）

| 顺序 | 任务 | 说明 |
|------|------|------|
| 2.1 | 初始拆解 | 调用模型 + 严格 JSON 解析；失败重试 1 次；仍失败则降级默认 JSON（见《Idea 拆解指令》）。 |
| 2.2 | 拆解校验 | 字段齐全、数组长度规则、`未明确` 标注规则等与拆解文档一致。 |
| 2.3 | 增量更新 | **每轮 3 题全部答完后**调用 1 次（共 3 次）：当前 JSON + 本轮 3 个 QA → 输出完整更新 JSON。 |

---

## 阶段 3：追问主循环（核心体验）

| 顺序 | 任务 | 说明 |
|------|------|------|
| 3.1 | Messages 组装 | `system` + `user(idea)` + `assistant(拆解JSON)` + 历史逐字问答对 + 最后 `user(当前追问生成指令)`。 |
| 3.2 | 九指令路由 | 按 `(round, question)` 选择《链式追问生成 Prompt》中对应模板并填参。 |
| 3.3 | 单题生成 | 模型只输出追问正文；生产环境不展示思考过程。 |
| 3.4 | 程序层质检 + Fallback | 长度、违禁词、问句完整性、用户上文引用、与历史问句重复度、轮次语气；不通过 → 修正指令重试 1 次 → 仍不通过 → 轮次模板追问（见链式追问文档 §5）。 |
| 3.5 | 轮次过渡文案 | R1-Q3 / R2-Q3 含进入下一轮预告；R3-Q3 含收束并引向盲区快照。 |

---

## 阶段 4：盲区与终局交付

| 顺序 | 任务 | 说明 |
|------|------|------|
| 4.1 | 盲区标记同次返回 | 生成追问时一并解析 `blindSpotMarker`（或 null）；解析失败不阻断追问。 |
| 4.2 | 会话内累积 | 维护 `blindSpotMarkers`：按维度去重/合并、严重程度取高（见《盲区快照生成 Prompt》§8.2）。 |
| 4.3 | 快照生成 | 9 问结束后：最终拆解 JSON + 累积 markers → 《盲区快照》Prompt → 解析结构化报告 JSON。 |
| 4.4 | 判定逻辑 | 代码实现 `getVerdict`（高风险数量规则），可与模型输出交叉校验，**以代码为最终展示依据更稳**。 |

---

## 阶段 5：前端与产品闭环（MVP）

| 顺序 | 任务 | 说明 |
|------|------|------|
| 5.1 | 入口与路由 | 工作台内进入压力测试、会话列表、继续/新建。 |
| 5.2 | 进行中 UI | 逐题一问一答、加载态、错误态（拆解降级可极简提示）。 |
| 5.3 | 结束页 | 拆解快照表、分级盲区清单、两档结论；操作：**再来一次 / 去成长教练 / 复制报告**。 |
| 5.4 | 与成长教练衔接 | 若需带盲区上下文，约定 query 或 session 传递的最小字段集。 |

---

## 阶段 6：质量与运维（可并行或后置）

| 顺序 | 任务 | 说明 |
|------|------|------|
| 6.1 | Layer2 异步评测 | 《追问质量评测体系》：自动化通过后约 10% 抽样 AI 打分，写日志，不阻塞用户。 |
| 6.2 | 回归用例集 | 品类用例 + 边界用例（空输入、乱码、英文、超长等）。 |
| 6.3 | Prompt 版本记录 | 版本号、修改说明、基线分、评测结果（先 Markdown/JSON 表亦可）。 |

## 里程碑建议

| 里程碑 | 包含阶段 | 验收标准 |
|--------|-----------|----------|
| **M1** | 0 + 1 + 2 | 输入 idea → 稳定产出拆解 JSON（可先控制台/调试面板验证）。 |
| **M2** | + 3（至少 R1 三问） | 第一轮链式追问跑通，含质检与 Fallback。 |
| **M3** | 3 全覆盖 | 9 问 + 每轮结束增量更新。 |
| **M4** | 4 + 5 | 盲区累积、快照页、判定、历史列表与结束页闭环。 |
| **M5** | 6 | 抽检评测与 Prompt 版本可追溯。 |
