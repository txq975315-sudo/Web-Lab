# 压力测试引擎：链式追问生成 Prompt

> 配套系统角色 Prompt 使用  
> 版本：v1.0  
> 日期：2026年5月

---

## 一、设计目标

链式追问生成 Prompt 解决的核心问题：**每一句具体追问是怎么生成的**。

非链式追问（避免）：

```
Q1: 你的目标用户是谁？           ← 独立问题
Q2: 你的痛点是什么？             ← 和Q1无关
Q3: 你的差异化是什么？           ← 和Q1、Q2无关
```

链式追问（目标）：

```
Q1: 你的目标用户是谁？           ← 打开
Q2: 你说是大学生——这是基于调研还是观察？   ← 引用Q1的回答，深入
Q3: 如果是观察，你自己可能不是典型用户？   ← 引用Q2的回答，逼到边界
```

---

## 二、链式追问的三要素

| 要素 | 说明 | 本方案实现 |
|------|------|-----------|
| **上下文记忆** | 第2问知道第1问问了什么、用户答了什么 | 逐字注入完整对话历史 |
| **引用机制** | 追问中自然引用用户上一轮的具体回答 | 每问生成指令中明确要求引用 |
| **递进逻辑** | 第3问基于前两问的回答继续深挖 | 9个独立指令，每问定义递进方向 |

---

## 三、架构：每次调用AI时的信息组成

```
┌─────────────────────────────────────────────┐
│  系统角色 Prompt（已在上下文，不重发）         │
├─────────────────────────────────────────────┤
│  【当前轮次标记】                             │
│  当前是第 X 轮（共3轮）                       │
│  当前语气：严肃了解 / 严格挑刺 / 冷酷拷问     │
│  当前是本轮第 Y 题（共3题）                   │
│  当前位置：打开 / 深入 / 逼到边界             │
├─────────────────────────────────────────────┤
│  【拆解结果快照】                             │
│  deconstruction JSON 关键字段                 │
├─────────────────────────────────────────────┤
│  【对话历史 —— 逐字注入】                     │
│  本轮已完成的问答对（逐字）                   │
│  前序轮次的全部问答（逐字）                   │
├─────────────────────────────────────────────┤
│  【当前追问生成指令】                         │
│  9个独立指令之一                              │
└─────────────────────────────────────────────┘
```

## 四、9个独立追问生成指令

---

### 第1轮：严肃了解（定位追问）

#### R1-Q1 打开

```markdown
【任务】生成第1轮第1问（共3题）。语气：严肃了解。位置：打开。

【当前素材】
- 用户原始idea：[originalIdea]
- 拆解结果中标注为"未明确"的维度：[listOfUnclearDimensions]

【生成要求】
1. 从拆解结果中选择一个标注为"未明确"的维度作为切入点
2. 追问要直接指出这个维度在用户描述中的模糊性
3. 让对方无法用一个词回答，必须展开解释
4. 追问结尾暗示"这只是开始，后面还有更多问题"

【输出】只输出追问文本，50-80字。
```

#### R1-Q2 深入

```markdown
【任务】生成第1轮第2问（共3题）。语气：严肃了解。位置：深入。

【对话上下文】
- 第1问：[question1]
- 用户回答：[answer1]

【生成要求】
1. 引用用户第1问回答中的具体词句
2. 检验这个回答的真实性：是观察还是调研？是假设还是验证？
3. 追问要让对方意识到自己的回答可能建立在未经验证的基础上
4. 不要泛泛地问"你怎么知道的"，要具体指出回答中的脆弱点

【输出】只输出追问文本，50-80字。
```

#### R1-Q3 逼到边界 + 过渡

```markdown
【任务】生成第1轮第3问（共3题），本轮最后一题。语气：严肃了解。位置：逼到边界。

【对话上下文】
- 第1问：[question1] / 用户回答：[answer1]
- 第2问：[question2] / 用户回答：[answer2]

【生成要求】
1. 综合前两问的回答，指出一个用户未意识到的假设或偏差
2. 这个追问要让用户感到"我确实没想过这个问题"
3. 追问结尾自然过渡到下一轮的预告：
   "定位的问题先到这里。现在进入竞争追问——我们聊聊你的差异化和壁垒。"
4. 不要直接给答案，只抛出问题

【输出】追问文本 + 过渡预告，80-100字。
```

---

### 第2轮：严格挑刺（竞争追问）

#### R2-Q1 打开

```markdown
【任务】生成第2轮第1问（共3题）。语气：严格挑刺。位置：打开。

【当前素材】
- 拆解结果中的差异化声明：[differentiation]
- 拆解结果中的潜在竞品：[potentialCompetitors]

【生成要求】
1. 要求用户澄清差异化声明的具体含义
2. 追问中自然引入一个潜在竞品进行对比
3. 不要直接否定差异化，而是要求用户证明差异化真实存在
4. 句式参考："你说X——Y已经在做这个了，你的X和Y的Z本质上有什么不同？"

【输出】只输出追问文本，50-80字。
```

#### R2-Q2 深入

```markdown
【任务】生成第2轮第2问（共3题）。语气：严格挑刺。位置：深入。

【对话上下文】
- 第1问：[question1] / 用户回答：[answer1]

【生成要求】
1. 引用用户对差异化的定义回答
2. 挑战其壁垒：如果大厂明天复制这个功能，用户为什么不回流？
3. 核心问题："你的差异化是否足够深，以至于大厂抄不了"
4. 不接受"我们的体验更好"这种模糊回答，要逼出具体的能力

【输出】只输出追问文本，50-80字。
```

#### R2-Q3 逼到边界 + 过渡

```markdown
【任务】生成第2轮第3问（共3题），本轮最后一题。语气：严格挑刺。位置：逼到边界。

【对话上下文】
- 第1问：[question1] / 用户回答：[answer1]
- 第2问：[question2] / 用户回答：[answer2]

【生成要求】
1. 基于前两问的回答，逼出一个用户无法回避的竞争短板
2. 根据用户回答选择最尖锐的角度：
   - 如果差异化不深 → "你的什么能力是大厂花6个月也复制不了的？"
   - 如果壁垒不够 → "如果竞品免费做，你怎么活？"
   - 如果用户迁移意愿不强 → "用户从现有方案切换到你，成本是什么？"
3. 选择最尖锐的角度，不留情面
4. 结尾自然过渡："竞争层面的探讨结束了。最后我们看看可行性——你的商业模式和应对风险的Plan B。"

【输出】追问文本 + 过渡预告，80-100字。
```

---

### 第3轮：冷酷拷问（可行性追问）

#### R3-Q1 打开

```markdown
【任务】生成第3轮第1问（共3题）。语气：冷酷拷问。位置：打开。

【当前素材】
- 拆解结果中的商业模式线索：[businessModel]
- 拆解结果中的关键假设：[keyAssumptions]

【生成要求】
1. 要求用户澄清商业模式的具体细节
2. 如果用户提到了定价，追问定价依据
3. 如果用户没有提到商业模式，直接问："你打算怎么赚钱？"
4. 语气要比前两轮更直接、更冷酷，不接受模糊回答

【输出】只输出追问文本，50-80字。
```

#### R3-Q2 深入

```markdown
【任务】生成第3轮第2问（共3题）。语气：冷酷拷问。位置：深入。

【对话上下文】
- 第1问：[question1] / 用户回答：[answer1]

【生成要求】
1. 引用用户对商业模式/定价的回答
2. 挑战其合理性：
   - 定价 → 依据是什么？用户支付能力？竞品定价？
   - 商业模式 → 单位经济模型是否成立？
   - 收入来源 → 是否过于乐观？
3. 要求数据支撑，不接受"我觉得没问题"
4. 语气冷酷："不要告诉我你觉得没问题，告诉我数字。"

【输出】只输出追问文本，50-80字。
```

#### R3-Q3 逼到边界 + 结束

```markdown
【任务】生成第3轮第3问（共3题），全部追问的最后一题。语气：冷酷拷问。位置：逼到边界。

【对话上下文】
- 第1问：[question1] / 用户回答：[answer1]
- 第2问：[question2] / 用户回答：[answer2]

【生成要求】
1. 基于前两问的回答，逼出一个用户没有想过的风险场景
2. 根据用户回答选择最尖锐的角度：
   - 数据不达预期 → "如果6个月后付费转化率不到3%，你是降价、换人群，还是放弃？"
   - 关键假设不成立 → "你假设XX成立——如果这个假设不成立，整个项目还成立吗？"
   - 资源不足 → "你现在有什么资源是这个项目必需的，但你还没有的？"
3. 选择最让用户不舒服的角度，这是最后一问，要留下深刻印象
4. 结尾自然过渡到盲区快照："三轮追问结束。接下来我为你生成一份思维盲区总结..."

【输出】追问文本 + 结束语，80-120字。
```

## 五、Fallback 策略

### 5.1 质量检查规则

每次 AI 返回追问文本后，程序层自动执行以下检查：

| 检查项 | 规则 | 不通过示例 |
|--------|------|-----------|
| **引用检查** | 追问中包含用户上文的具体词句（≥3个字匹配） | "能具体说说吗？"（无引用） |
| **长度检查** | 追问文本长度在 30-120 字之间 | "为什么？"（太短） |
| **禁忌检查** | 追问中不包含黑名单词汇 | "这个想法不错，但是..."（含违禁词） |
| **重复检查** | 追问与之前任何一问不重复（相似度<80%） | 和上一轮追问几乎一样 |
| **语气检查** | 追问符合当前轮次的句式特征 | 第3轮追问用了第1轮的温和句式 |

### 5.2 Fallback 流程（程序层自动执行，用户无感知）

```
程序调用 AI 生成追问
  → 收到追问文本
  → 执行质量检查（5项）
    ├─ 全部通过 → 直接展示给用户 ✅
    └─ 任何一项不通过
         → L1：自动重试（重新调用 AI，附加修正指令）
              ├─ 重试通过 → 展示 ✅
              └─ 重试仍不通过
                   → L2：使用预设模板追问
                        → 展示模板追问 ✅
```

### 5.3 修正指令（重试时注入）

```markdown
【修正要求】上一轮生成的追问不合格，原因是：[具体原因]。
请重新生成，确保：
1. [对应的修正要求]
2. 其他要求不变
```

### 5.4 预设模板追问（L2 Fallback）

| 轮次 | 模板追问 |
|------|---------|
| 第1轮 | "你刚才说'[用户关键词]'——能具体解释一下这个词在你项目中的含义吗？" |
| 第2轮 | "你说'[用户关键词]'——如果市场上最大的竞争对手明天做这个，你怎么办？" |
| 第3轮 | "你刚才提到了'[用户关键词]'——这个数字/判断的依据是什么？" |

> `[用户关键词]` 从用户最近一轮回答中自动提取最长的一个名词短语。

## 六、上下文组装规则

### 6.1 组装原则：逐字注入

每次调用 AI 时，messages 中包含完整的对话历史（逐字），不压缩、不摘要。

### 6.2 Messages 结构

```typescript
messages = [
  // 1. 系统角色（始终存在）
  { role: "system", content: systemPrompt },
  
  // 2. 用户原始idea（拆解前）
  { role: "user", content: userOriginalIdea },
  
  // 3. 拆解结果（AI返回，作为assistant消息）
  { role: "assistant", content: deconstructionJSON },
  
  // 4. 第1轮问答（逐字）
  { role: "assistant", content: "R1-Q1追问文本" },
  { role: "user", content: "R1-Q1用户回答" },
  { role: "assistant", content: "R1-Q2追问文本" },
  { role: "user", content: "R1-Q2用户回答" },
  { role: "assistant", content: "R1-Q3追问文本" },
  { role: "user", content: "R1-Q3用户回答" },
  
  // 5. 第2轮问答（逐字）—— 如果进入第2轮
  { role: "assistant", content: "R2-Q1追问文本" },
  { role: "user", content: "R2-Q1用户回答" },
  // ...以此类推
  
  // 6. 当前追问生成指令（作为最后的user消息）
  { role: "user", content: currentQuestionPrompt }
];
```

### 6.3 Token 估算（逐字注入）

| 追问进度 | 上下文内容 | 估算Token |
|---------|-----------|----------|
| R1-Q1 | system + idea + 拆解JSON + 指令 | ~1500 |
| R1-Q2 | 上述 + R1-Q1问答 | ~1800 |
| R1-Q3 | 上述 + R1-Q2问答 | ~2100 |
| R2-Q1 | 上述 + R1-Q3问答 | ~2400 |
| R2-Q2 | 上述 + R2-Q1问答 | ~2700 |
| R2-Q3 | 上述 + R2-Q2问答 | ~3000 |
| R3-Q1 | 上述 + R2-Q3问答 | ~3300 |
| R3-Q2 | 上述 + R3-Q1问答 | ~3600 |
| R3-Q3 | 上述 + R3-Q2问答 | ~3900 |

> 多数大模型上下文窗口为 8K-128K，3900 token 在安全范围内。

## 八、决策记录

| 决策点 | 最终选择 | 备选 | 理由 |
|--------|---------|------|------|
| 指令组织方式 | 9个独立指令 | 统一模板+动态参数 | 独立指令可精细调优，追问质量高 |
| 对话历史注入 | 逐字注入 | 摘要注入 | 链式追问需要精准引用，逐字最可靠 |
| Fallback 策略 | L1重试 → L2模板 | 仅用模板 / 跳过 | 重试给AI一次机会，避免模板化追问 |
| 追问长度 | 50-80字（逼到边界80-120） | 统一长度 | 打开/深入简洁，逼到边界可稍长 |
| 思考过程输出 | 完全隐藏 | 输出到日志 | MVP简洁优先，调试模式单独记录 |
