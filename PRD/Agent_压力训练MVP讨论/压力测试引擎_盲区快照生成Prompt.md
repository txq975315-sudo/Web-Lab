# 压力测试引擎：盲区快照生成 Prompt

> 配套系统角色 Prompt + 链式追问 + 拆解指令使用  
> 版本：v1.0  
> 日期：2026年5月

---

## 一、盲区快照的定位

盲区快照是压力测试的**终端交付物**。用户在完成 9 轮追问后，看到的一份结构化思维诊断报告。

**核心目标**：让用户清楚地知道——
1. AI 是如何理解我的 idea 的（拆解快照）
2. 我的 idea 有哪些盲区（盲区清单）
3. 我的 idea 整体处于什么水平（判定结果）

---

## 二、输入数据

生成盲区快照需要以下数据：

| 数据源 | 内容 | 用途 |
|--------|------|------|
| **最终拆解 JSON** | 9维度结构化信息（含迭代更新结果） | 展示 AI 对 idea 的理解 |
| **全部 9 轮对话** | 逐字的追问+回答 | 分析盲区 |
| **追问质量记录** | 每问用户回答的"模糊度评分" | 量化盲区严重程度 |

---

## 三、盲区分析逻辑

### 3.1 盲区来源（3层）

```
盲区 = 拆解盲区（维度缺失）+ 对话盲区（回答暴露）+ 假设盲区（未验证前提）
```

| 盲区类型 | 发现方式 | 示例 |
|---------|---------|------|
| **拆解盲区** | 拆解结果中标注"未明确" | targetUser="未明确" |
| **对话盲区** | 用户在回答中暴露的逻辑漏洞 | "我觉得没问题"（无数据） |
| **假设盲区** | 关键假设未经用户回答验证 | "用户愿意为专注付费" |

### 3.2 盲区提取方法

盲区不是由 AI 在快照阶段"凭空分析"出来的，而是在**追问过程中实时积累**的：

```
追问过程中：
  R1-Q1 → 用户回答模糊 → 标记：targetUser定义不清
  R1-Q2 → 用户承认是观察 → 标记：缺乏用户调研
  R2-Q1 → 用户无法回答差异点 → 标记：差异化站不住
  ...（每轮追问中持续积累）

快照阶段：
  → 汇总所有已标记的盲区
  → 去重、分级、组织
  → 生成最终报告
```

**关键设计**：追问过程中实时积累，快照阶段只做汇总，不做新的分析。

### 3.3 追问过程中的盲区标记

每个追问生成指令中包含**盲区标记指令**：

```markdown
【追问生成后】
请同时输出一个"盲区标记"（JSON格式）：
{
  "blindSpotFound": true/false,  // 本轮追问是否暴露了新的盲区
  "dimension": "目标用户/痛点/差异化/商业模式",  // 盲区所属维度
  "description": "具体描述",  // 盲区的文字描述
  "severity": "high/medium/low"  // 严重程度
}

严重程度判断：
- high：用户无法回答，或回答自相矛盾，或核心假设不成立
- medium：用户回答模糊，或缺乏依据，或非核心维度有问题
- low：用户回答了但有优化空间，或次要维度不够完善
```

---

## 四、严重程度判定标准

### 4.1 三个级别定义

| 级别 | 图标 | 定义 | 对用户的影响 |
|------|------|------|-------------|
| **高** | 🔴 | 核心维度存在根本性缺陷，idea 成立的前提不成立 | 必须重新思考该维度 |
| **中** | 🟡 | 核心维度有模糊点，但 idea 仍有讨论空间 | 需要补充验证或调整 |
| **低** | 🟢 | 非核心维度不够完善，不影响 idea 整体方向 | 建议优化但非必须 |

### 4.2 判定矩阵

| 维度 | high | medium | low |
|------|------|--------|-----|
| **目标用户** | 未定义/定义自相矛盾 | 定义笼统无场景 | 场景可再细分 |
| **痛点** | 痛点不存在/虚构 | 痛点真实但未被验证 | 痛点多场景覆盖不全 |
| **差异化** | 无差异化/差异化已被竞品覆盖 | 差异化存在但壁垒不够 | 差异化可再深化 |
| **商业模式** | 无商业模式/单位经济不成立 | 商业模式有但定价无依据 | 收入渠道可再拓展 |
| **关键假设** | 核心假设不成立 | 假设未验证 | 假设有弱验证 |

## 五、最终判定逻辑

### 5.1 判定规则

```typescript
function getVerdict(blindSpots: BlindSpotItem[]): 'needs_rethink' | 'basically_viable' {
  const highRiskCount = blindSpots.filter(s => s.severity === 'high').length;
  const mediumRiskCount = blindSpots.filter(s => s.severity === 'medium').length;
  
  if (highRiskCount >= 2) return 'needs_rethink';        // 2+ 高风险 → 建议深入思考
  if (highRiskCount >= 1 && mediumRiskCount >= 2) return 'needs_rethink';  // 1高+2中
  return 'basically_viable';                              // 否则 → 基本可行
}
```

### 5.2 判定结果展示

| 结果 | 展示文案 | 含义 |
|------|---------|------|
| **🔴 建议深入思考** | "关键维度存在未验证假设，建议在以下方向补强后再评估" | 核心逻辑有硬伤 |
| **🟢 基本可行** | "核心逻辑通顺，仍有优化空间，建议针对以下盲区持续迭代" | 方向没问题，细节需打磨 |

## 六、快照生成 Prompt

```markdown
【任务】基于本轮追问过程中积累的所有盲区标记，生成最终的思维盲区快照。

【输入数据】
1. 最终拆解 JSON：{finalDeconstructionJSON}
2. 盲区标记列表（追问过程中实时积累）：{blindSpotMarkers}

【生成步骤】
步骤1：汇总
- 将所有盲区标记按维度归类（目标用户/痛点/差异化/商业模式）
- 同一维度的盲区去重合并

步骤2：分级
- 对每个盲区确认严重程度（high/medium/low）
- 如果同一维度有多个盲区，取最高级别

步骤3：判定
- 计算 high 和 medium 数量
- 根据判定规则输出"建议深入思考"或"基本可行"

步骤4：组织输出
按以下格式输出：

【拆解快照】
用表格展示AI对idea的理解（维度 × 用户声明 × AI追问发现）

【盲区清单】
按严重程度排序（🔴高 → 🟡中 → 🟢低），每个盲区包含：
- 所属维度
- 盲区描述
- 严重程度
- 修复建议

【判定结果】
"建议深入思考"或"基本可行" + 一句话解释

【输出格式】
严格的JSON格式：

{
  "deconstructionSnapshot": [
    { "dimension": "目标用户", "userClaim": "大学生", "aiFinding": "⚠️ 场景未细分" },
    ...
  ],
  "blindSpots": [
    { "dimension": "商业模式", "description": "定价缺乏依据", "severity": "high", "suggestion": "调研目标用户支付意愿和竞品定价" },
    ...
  ],
  "verdict": "needs_rethink",
  "verdictText": "关键维度存在未验证假设...",
  "totalHigh": 2,
  "totalMedium": 1,
  "totalLow": 1
}
```

## 七、快照展示格式（前端渲染参考）

### 7.1 拆解快照表格

| 维度 | 你的描述 | AI 的追问发现 |
|------|---------|--------------|
| 目标用户 | 大学生 | ⚠️ 场景未细分（备考/社交/求职） |
| 痛点 | 专注力被碎片化App毁掉 | ⚠️ 未验证是观察还是调研 |
| 差异化 | 建筑感空间 | ✅ 有独特定义 |
| 商业模式 | 订阅制 | ❌ 定价逻辑不充分 |
| 关键假设 | 大学生愿意为专注付费 | ❌ 未考虑支付能力 |

图标说明：✅ 通过 / ⚠️ 有盲区 / ❌ 高风险

### 7.2 盲区清单

🔴 高风险（2个）
1. **商业模式 · 定价缺乏依据** —— 建议调研目标用户支付意愿和竞品定价
2. **关键假设 · 支付能力未验证** —— 建议至少访谈10个目标用户确认付费意愿

🟡 中风险（1个）
3. **目标用户 · 场景未细分** —— 建议明确首个细分场景（如考研），集中资源验证

🟢 低风险（1个）
4. **痛点 · 未验证真实性** —— 建议通过问卷或访谈验证痛点普遍存在

### 7.3 判定结果

🔴 **建议深入思考** —— 关键维度存在未验证假设，建议在商业模式和用户验证方向补强后再评估。

### 7.4 操作按钮

- [🔁 再来一次] —— 重新开始一轮压力测试
- [📚 去成长教练] —— 针对盲区进行专项学习
- [📋 复制报告] —— 复制完整报告文本

## 八、追问过程中的盲区标记实现

### 8.1 每问生成时的盲区标记

```typescript
// 生成追问时，同时返回盲区标记
interface QuestionGenerationResult {
  questionText: string;       // 追问文本（展示给用户）
  blindSpotMarker?: {         // 盲区标记（不展示，后台积累）
    dimension: string;
    description: string;
    severity: 'high' | 'medium' | 'low';
  };
}

// 追问生成 Prompt 中追加：
// 【盲区标记】请基于用户的回答，判断本轮是否暴露了新的盲区。
// 如果暴露了，输出盲区标记JSON；如果没有，输出 null。
```

### 8.2 盲区标记的积累

```typescript
// 会话状态中维护盲区列表
interface PressureSession {
  // ...其他字段
  blindSpotMarkers: BlindSpotMarker[];  // 追问过程中实时积累
}

// 每轮追问后追加
function recordBlindSpot(session: PressureSession, marker: BlindSpotMarker) {
  // 去重：同一维度的盲区合并
  const existing = session.blindSpotMarkers.find(m => m.dimension === marker.dimension);
  if (existing) {
    // 取严重程度更高的
    existing.severity = getHigherSeverity(existing.severity, marker.severity);
    existing.description = mergeDescription(existing.description, marker.description);
  } else {
    session.blindSpotMarkers.push(marker);
  }
}
```

## 九、决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 盲区分析时机 | **追问过程中实时积累** | 快照阶段只做汇总，不做新分析，保证一致性 |
| 严重程度 | **3级（高/中/低）** | 足够区分优先级，不复杂 |
| 判定结果 | **2档（建议深入思考/基本可行）** | 简洁明了，不多级评分 |
| 快照格式 | **表格+清单+判定** | 结构化展示，一目了然 |
| 操作按钮 | **再来一次/成长教练/复制报告** | 形成完整闭环 |
| 追问生成附加 | **同步返回盲区标记** | 不增加额外AI调用 |
