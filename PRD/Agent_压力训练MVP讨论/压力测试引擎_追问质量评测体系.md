# 压力测试引擎：追问质量评测体系 + Prompt 版本管理系统

> 配套系统角色 Prompt + 链式追问 + 拆解指令 + 盲区快照使用  
> 版本：v1.0  
> 日期：2026年5月

---

## 一、设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 评测方案 | **10%抽样AI评测 + 程序层硬性检查** | MVP平衡质量与效率，生产环境零额外延迟 |
| 通过线 | **60分（中等）** | 平衡质量和效率，不触发过多Fallback |
| Prompt版本管理 | **建立独立系统** | 每次修改Prompt都记录评测基线，支持持续优化 |

---

## 二、追问质量 = 5维度评分

```
总分 = 引用质量×30% + 语气准确性×25% + 问题深度×25% + 文本质量×10% + 独特性×10%
通过线：总分 ≥ 60分，且每个维度 ≥ 2分
```

### 维度1：引用质量（30%）

| 分数 | 标准 | 示例 |
|------|------|------|
| **5分** | 精准引用用户上文具体词句（≥5字），自然嵌入 | "你说'建筑感空间'——Forest已经做了8年…" |
| **4分** | 引用了用户上文但较笼统（2-4字） | "你说你的差异化…" |
| **3分** | 只泛泛提到"你刚才说"，无具体词句 | "你刚才说的那个…" |
| **1分** | 完全没有引用上文 | "能具体说说吗？" |

### 维度2：语气准确性（25%）

| 分数 | 第1轮（严肃了解） | 第2轮（严格挑刺） | 第3轮（冷酷拷问） |
|------|------------------|------------------|------------------|
| **5分** | "你说X——X的定义是什么？" | "你说X——如果大厂做呢？" | "你凭什么认为…？告诉我数字。" |
| **3分** | 用了第2轮句式 | 用了第3轮句式 | 用了第1轮句式 |
| **1分** | 用了第3轮句式 | 用了第1轮句式 | 用了第1轮句式 |

### 维度3：问题深度（25%）

| 分数 | 标准 | 示例 |
|------|------|------|
| **5分** | 无法用一个词回答，必须展开；暴露思维盲区 | "如果Forest明天加'虚拟书房'，你的用户会流失吗？什么能力是Forest6个月复制不了的？" |
| **3分** | 无法用一个词回答，但深度一般 | "你的差异化具体指什么？" |
| **1分** | 可以用一个词或一句话搪塞 | "为什么？" / "能具体说说吗？" |

### 维度4：文本质量（10%）

| 检查项 | 通过标准 | 失败处理 |
|--------|---------|---------|
| 长度 | 30-120字 | 程序层拦截，触发Fallback |
| 违禁词 | 不包含黑名单词汇 | 程序层拦截，触发Fallback |
| 语病 | 无明显语法错误 | AI评测标记 |
| 完整性 | 完整问句，非片段 | 程序层拦截 |

### 维度5：独特性（10%）

| 分数 | 标准 |
|------|------|
| **5分** | 与之前所有问题相似度<30%，有独特切入角度 |
| **3分** | 相似度30-60%，有一定新意但有重复结构 |
| **1分** | 相似度>60%，几乎重复 |

## 三、3层评测体系

```
自动化评测（程序层）→ 快速筛选，零延迟
  ↓ 通过
半自动化评测（AI评测）→ 质量评分，后台异步
  ↓ 低分
人工评测（专家抽样）→ 最终校准，持续优化
```

### Layer 1：自动化评测（程序层硬性检查）

每个追问生成后立即执行，**不通过则触发 Fallback**：

```typescript
function autoEvaluate(question: string, context: EvalContext): AutoScore {
  const checks = {
    lengthOk: question.length >= 30 && question.length <= 120,
    noForbiddenWords: !hasForbiddenWords(question),
    noGrammarError: !hasGrammarError(question),
    isCompleteSentence: isQuestion(question),
    referencesUserText: hasReference(question, context.userLastAnswer),
    notDuplicate: !isDuplicate(question, context.previousQuestions),
  };
  
  return {
    passed: Object.values(checks).every(Boolean),
    failedChecks: Object.entries(checks)
      .filter(([, v]) => !v)
      .map(([k]) => k),
  };
}
```

### Layer 2：半自动化评测（AI评测，10%抽样）

自动化通过后，**10%的追问进入AI评测队列**，后台异步执行：

```markdown
【角色】追问质量评审专家
【任务】对以下追问从5个维度评分（1-5分）
【追问文本】{questionText}
【当前轮次】第{X}轮
【用户上文】{userLastAnswer}
【之前的问题】{previousQuestions}
【输出格式】
{
  "scores": {
    "referenceQuality": 1-5,
    "toneAccuracy": 1-5,
    "questionDepth": 1-5,
    "textQuality": 1-5,
    "uniqueness": 1-5
  },
  "totalScore": 0-100,
  "passed": true/false,
  "feedback": "改进建议"
}
```

**总分计算**：
```
totalScore = referenceQuality×30 + toneAccuracy×25 + questionDepth×25 + textQuality×10 + uniqueness×10
通过线：totalScore ≥ 60，且每个维度 ≥ 2
```

### Layer 3：人工评测（专家抽样）

**触发条件**：
- 每周随机抽取 10% 的追问
- AI 评分 ≤ 3 分的追问强制人工复查
- 新 Prompt 版本上线前，必须完成一轮完整人工评测

**人工评测表**：

| 追问ID | 轮次 | 引用 | 语气 | 深度 | 文本 | 独特 | 总分 | 通过 | 问题描述 |
|--------|------|------|------|------|------|------|------|------|---------|
| R1Q1 | 1/1 | 5 | 4 | 4 | 5 | 5 | 88 | ✅ | - |
| R2Q2 | 2/2 | 2 | 3 | 3 | 5 | 4 | 52 | ❌ | 未引用用户差异化回答 |

## 五、Prompt 版本管理系统

### 5.1 核心功能

每次修改 Prompt 时，记录：
- 版本号 + 修改日期 + 修改人
- 修改内容（diff）
- 修改理由
- **评测基线**（修改前的平均分）
- **评测结果**（修改后的平均分）
- 是否通过评测（修改后平均分 ≥ 基线）

### 5.2 版本记录表

| 版本 | 日期 | 修改内容 | 基线分 | 新分 | 通过 | 备注 |
|------|------|---------|--------|------|------|------|
| v1.0 | 2026-05-01 | 初始版本 | - | 72 | ✅ | 首次评测 |
| v1.1 | 2026-05-08 | 增加第3轮语气强度 | 72 | 78 | ✅ | 明显提升 |
| v1.2 | 2026-05-15 | 修改引用原则要求≥5字 | 78 | 65 | ❌ | 过度严格，回滚 |
| v1.3 | 2026-05-20 | 优化Few-shot示例 | 78 | 82 | ✅ | 示例更有效 |

### 5.3 评测驱动优化流程

```
发现低分追问模式
  → 分析共同原因（哪个Prompt维度有问题）
  → 修改对应Prompt
  → 用测试用例集重新评测
  → 对比基线分数
    ├─ 分数提升 ≥ 5分 → 发布新版本 ✅
    ├─ 分数持平 → 观察一周再决定
    └─ 分数下降 → 回滚到上一版本 ❌
```

### 5.4 代码层实现

```typescript
interface PromptVersion {
  version: string;           // "v1.0"
  createdAt: number;         // 时间戳
  author: string;            // 修改人
  diff: string;              // 修改内容描述
  reason: string;            // 修改理由
  baselineScore: number;     // 修改前基线分
  newScore: number;          // 修改后新分
  passed: boolean;           // 是否通过评测
  testResults: TestResult[]; // 详细测试结果
}

interface PromptVersionManager {
  // 创建新版本
  createVersion(changes: PromptChanges): Promise<PromptVersion>;
  
  // 运行评测
  runEvaluation(version: string, testCases: TestCase[]): Promise<TestResult[]>;
  
  // 对比两个版本
  compareVersions(v1: string, v2: string): ComparisonResult;
  
  // 回滚到指定版本
  rollbackTo(version: string): void;
  
  // 获取评测趋势
  getScoreTrend(): ScoreDataPoint[];  // 用于图表展示
}
```

## 六、决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 评测方案 | 10%抽样AI + 程序层硬性检查 | MVP平衡质量与效率，零额外延迟 |
| 通过线 | 60分，每维度≥2 | 平衡质量和效率 |
| 评测维度 | 5维度（引用/语气/深度/文本/独特） | 覆盖追问质量全部关键方面 |
| 权重分配 | 30/25/25/10/10 | 引用最重要，文本和独特为辅 |
| AI评测时机 | 后台异步，不阻塞追问 | 用户体验优先 |
| 人工评测 | 每周10% + 低分强制复查 + 版本上线前 | 三重保障 |
| Prompt版本管理 | 独立系统，评测驱动 | 每次修改有据可依 |
| 版本通过标准 | 新分≥基线+5才发布 | 防止负优化 |
