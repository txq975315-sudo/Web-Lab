## 🎯 一、需求背景

### 1.1 用户反馈的核心问题

用户在使用过程中发现了 **4 个关键问题**：

#### ❌ **问题 1：导航栏认知过载**
```
旧版导航栏预置了 22+ 个固定模板节点：
├─ 01 项目宪法（8 个子节点，大部分为空）
├─ 02 市场与用户洞察（6 个子节点，全部为空）
├─ 03 产品与商业策略（5 个子节点，全部为空）
├─ 04 决策链图谱
├─ 05 反脆弱审计（3 个子节点，全部为空）
└─ 06 执行路线图（4 个子节点，全部为空）

问题：
- 大量空白节点占用视觉空间
- 用户无法区分哪些是"已有内容"vs"空白模板"
- 认知负荷高：需要理解 22+ 个模板名称
```

#### ❌ **问题 2：中部 ArchivePanel 联动失效**
```
现象：
- 点击 Sidebar 的任意文档节点
- 中部 ArchivePanel 永远显示列表视图
- 无法查看/编辑单个文档的详情

原因：
- ArchivePanel 未根据 activeDocId 做条件渲染
- 文档查找逻辑不完善
```

#### ❌ **问题 3：顶部核心定位区域未更新**
```
现象：
- 显示固定的"未设置核心定位"文字
- 点击无反应或反应不符合预期
- 未从项目树动态读取真实数据

期望：
- 显示 manifesto 文档的真实 slogan 字段
- 点击后打开核心定位编辑页面
```

#### ❌ **问题 4：空模块完全隐藏**
```
现象：
- 没有文档的模块直接从导航栏消失
- 用户无法在这些模块下创建新内容
- 违反"可发现性"原则

用户原话：
"下面隐藏的怎么查看呢？直接就没有了"
```

### 1.2 用户提出的解决方案

用户给出了清晰的 **架构转型方向**：

> **从"推式架构"到"拉式架构"的转变**
>
> - **导航栏应该是"资产仓库的目录"，不是"施工图纸的货架"**
> - **模板是后台的生产工具，不应该占用导航栏的认知空间**
> - **每个导航节点都应该是用户亲手创建的有意义的文档**

```javascript
  {

  },

// 2. 动态计算文档数量
const getModuleDocuments = useCallback((moduleConfig) => {
  const category = activeProject.children.find(cat => 
    cat.categoryType === moduleConfig.categoryType || 
    cat.id.split('-').pop() === moduleConfig.id
  )
  return category?.children?.filter(doc => doc.type === 'document') || []
}, [activeProject])

// 3. 过滤可见模块
const visibleModules = useMemo(() => {
  return MODULE_STRUCTURE.filter(moduleConfig => {
    if (moduleConfig.isAlwaysVisible) return true
    return getModuleDocuments(moduleConfig).length > 0
  })
}, [activeProject, getModuleDocuments])
```

### 3.2 关键技术决策

#### 决策 1：为什么用 `categoryType` 而不是硬编码 ID？

**选项对比**：

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| A. 硬编码 ID (`cat-constitution`) | 简单直接 | 脆弱，ID 变化就坏 | ❌ |
| B. `categoryType` 元数据 | 灵活，语义清晰 | 需要初始化时正确设置 | ✅ |
| C. 正则匹配 | 最灵活 | 性能差，难维护 | ❌ |

**最终选择 B**：在 `createProject()` 时为每个分类设置 `categoryType`，后续通过该字段匹配。

---

#### 决策 2：为什么保留 `isAlwaysVisible` 而不是全部动态？

**考虑因素**：

1. **宪法模块**：即使没有文档（理论上不可能），也应该始终可见作为锚点
2. **决策链图谱**：特殊视图，不是普通文档容器
3. **用户体验**：避免导航栏突然变空导致迷失

**结论**：混合模式 - 大部分动态，关键模块强制显示。

---

#### 决策 3：为什么清理 console.log？

**性能影响分析**：

```javascript
// 修改前（每次渲染执行）
console.log('📦 ArchivePanel 渲染:', { 
  activeDocId, 
  hasProject: !!activeProject, 
  allDocsCount: allDocuments.length  // 对象序列化开销
})

// 在 useMemo 中（每次依赖变化执行）
console.log('🎯 ConstitutionAnchor 计算 displaySlogan:', {
  hasConstitution: !!constitution,
  manifestoFields: constitution?.manifesto?.fields,  // 大对象
  corePositioning: constitution?.corePositioning,
  hasActiveProject: !!activeProject
})
```

**实测影响**：
- 渲染频率：~60fps（每帧都输出）
- 内存占用：对象序列化 + 字符串拼接
- 结果：`ERR_ABORTED` 错误（浏览器中止加载）

**解决**：移除所有非错误处理的日志，仅保留 `console.error`。

1. `constitution.manifesto.fields.slogan`

#### `handleCreateFromTemplate(template, moduleId)`

**位置**：`Sidebar.jsx:793-857`

**功能**：从模板创建新文档的完整流程

**流程**：
```
1. 解析目标模块 ID（自动推荐或手动指定）
2. 在项目树中找到目标分类节点
3. 创建新文档对象（含默认字段）
4. 调用 LabContext.addDocument()
5. 展开目标模块（setExpandedModules）
6. 选中新文档（setActiveDocId）
7. 打开详情页（延迟 200ms 确保状态更新）
```

**错误处理**：
- 找不到分类 → alert 提示
- addDocument 失败 → try-catch + alert
- addDocument 不存在 → alert 系统错误

#### 问题 2：点击 Sidebar 无反应

**现象**：
- 点击文档节点，中部不变
- 点击模块标题，中部不变

#### 问题 3：ConstitutionAnchor 显示"未设置核心定位"

**现象**：
- 即使 manifesto 文档已创建并有 slogan
- 顶部仍显示"未设置核心定位"

**原因**：查找逻辑单一

```javascript
// 旧版：只从一个来源读取
const displaySlogan = constitution.manifesto?.fields?.slogan 
  || constitution.corePositioning 
  || '未设置核心定位'
```

**问题**：
- 如果 `constitution.manifesto` 为 undefined → 直接跳到 corePositioning
- 如果 `corePositioning` 也是默认值 → 显示"未设置核心定位"
- 但实际上项目树中的 manifesto 文档可能有真实数据！

**解决方案**：三级查找 + 项目树回退

```javascript
const displaySlogan = useMemo(() => {
  // Level 1: constitution.manifesto.fields.slogan
  const s1 = constitution?.manifesto?.fields?.slogan
  if (s1?.trim()) return s1
  
  // Level 2: constitution.corePositioning
  const s2 = constitution?.corePositioning
  if (s2 && s2 !== '未设置核心定位') return s2
  
  // Level 3: 项目树 manifesto 文档 ← 新增！
  if (activeProject?.children) {
    const cat = activeProject.children.find(c => 
      c.categoryType === 'constitution'
    )
    const doc = cat?.children?.find(d => d.docType === 'manifesto')
    if (doc?.fields?.slogan?.trim()) return doc.fields.slogan
  }
  
  return '未设置核心定位'
}, [constitution, activeProject])
```

#### ✅ 好的做法

1. **渐进式开发**：先实现核心功能，再优化细节
2. **详细日志**：开发阶段使用日志追踪数据流（上线前清理）

5. **Git 小步提交**：每个功能点完成后立即 commit

#### ⚠️ 需要注意的坑

1. **Console.log 性能陷阱**：不要在渲染函数/ useMemo 中输出大对象
2. **useMemo 依赖完整性**：漏掉依赖会导致缓存失效
3. **状态更新时机**：setState 后立即读取可能还是旧值（用 setTimeout 或 useEffect）
4. **ID 格式一致性**：确保生成和查找时使用相同的 ID 格式

### 9.1 设计原则参考

1. **渐进式披露 (Progressive Disclosure)**
   - 只在需要时显示详细信息
   - 应用：空模块灰色折叠，展开后才显示模板选择器

2. **拉式架构 (Pull-based Architecture)**
   - 用户主动获取所需功能
   - 应用：模板退居后台，按需调用

3. **原子设计 (Atomic Design)**
   - Atoms → Molecules → Organisms → Templates → Pages
   - 应用：DocumentDetail (Organism) 由多个 Field (Atom) 组成
