## 1. 模块概述

### 产品定位

商业假设构建器（Business Hypothesis Builder）是 Thinking Lab 中「练」标签下的第二个子模式，与压力测试完全独立。其核心定位如下：

- **独立引擎**：与压力测试引擎无代码耦合、无数据耦合。两者共享同一入口标签（「练」），但内部通过子模式切换（liveSubMode）隔离渲染和逻辑。
- **4 轮 x 2 问**：共计 8 轮 AI 追问引导，覆盖用户与痛点、解决方案、商业模式、验证计划四个维度。每轮 2 问，用户逐轮回答，所有 8 问答完后自动生成商业假设画布。
- **商业教练角色**：AI 以温和共建式的「商业教练」角色引导用户，而非压力测试的「压力测试官」角色。核心原则是「不评判、不打分、只引导」，帮助用户把模糊想法变成结构化假设。
- **输出物**：商业假设画布（Business Canvas），包含用户假设、解决方案假设、商业模式假设、验证计划（P0/P1/P2 三级优先级）和下一步行动。

### 与压力测试的根本区别

| 维度 | 压力测试 | 商业假设构建器 |
|------|---------|---------------|
| 产品目标 | 检验已有方案的漏洞与盲区 | 从模糊想法逐步梳理成可执行的商业假设 |
| AI 角色 | 压力测试官（挑剔、批判、层层追问） | 商业教练（温和、共建、引导） |
| 对话结构 | 1 轮拆解 + 9 问追问 + 盲区报告 | 4 轮 x 2 问，共 8 问 + 商业假设画布 |
| 初始步骤 | 有 AI 初始拆解（deconstructing） | 无初始拆解，直接进入 R1Q1 |
| 输出物 | 盲区快照 / 盲区报告 | 商业假设画布 + 验证计划表 |
| 质检维度 | 压力引擎专属质检（如对抗性、遗漏率） | 合作共建式质检（如语气、引用、长度） |
| 初始状态 | `deconstructing`（拆解中） → `questioning` | `ideating`（构思中） → `questioning` |
| 数据存储 | `thinking-lab-pressure-engine-sessions` | `thinking-lab-hypothesis-sessions` |

## 2. 架构设计：Feature 模式

商业假设构建器完整复刻压力测试引擎的多层 Feature 架构，由以下 7 层组成：

```
├── hypothesisQC.js             # QC — AI 输出质检（Layer 1 + Fallback）
├── hypothesisEvalLog.js        # EvalLog — 抽检与运维日志
├── BusinessCanvasViewer.jsx    # UI — 商业假设画布展示组件

├── HypothesisSessionInlineViewer.jsx # UI — 历史会话内联查看器

```

对应的 Prompt 配置文件位于：

```

```

### 各层职责

| 层        | 文件                     | 职责 |
|-----------|--------------------------|------|
| Types     | `hypothesisTypes.js`     | 定义 HypothesisSession、HypothesisRound、HypothesisQuestion、BusinessCanvas 等数据契约，以及 createEmptyRounds、findAwaitingAnswerSlot、allQuestionsAnswered 等纯函数 |
| Store     | `hypothesisStore.js`     | localStorage 的 CRUD 操作，使用独立 Bucket key，通过 CustomEvent 通知变更 |
| Api       | `hypothesisApi.js`       | 封装 AI 调用管线：bootstrapHypothesisSession（首问生成）、generateHypothesisQuestion（单问生成）、submitHypothesisAnswer（回答提交+自动推进）、runCanvasGeneration（画布生成） |
| QC        | `hypothesisQC.js`        | Layer 1 质检：评估追问的长度、违禁词、问句完整性、共建语气、引用、重复度；失败后触发 Retry 或模板 Fallback；提供 templateFallbackQuestion / buildRetryHint |
| EvalLog   | `hypothesisEvalLog.js`   | 环形缓冲日志（最大 200 条），记录 Layer1 QC 结果和 Layer2 抽检占位 |
| UI        | 6 个组件文件              | 入口工作台、会话运行器、内联查看器、历史面板、进度条、画布展示器 |

## 3. 入口集成方式

### 顶层路由

用户通过顶栏「练」标签进入 `ModuleSegmentedControl`（`src/components/workbench/ModuleSegmentedControl.jsx`），该控件提供三个标签页：

- **练（live）** — 压力测试 / 假设构建子模式切换
- **学（coach）** — 成长教练
- **看（archaeology）** — 对话考古

### 子模式切换

在 `src/components/LabPanel.jsx` 中，`live` 模式下通过 `liveSubMode` 状态（sessionStorage 持久化 key: `thinking-lab-live-sub-mode`）实现子模式切换：

```jsx
// 子模式切换按钮（LabPanel.jsx lines 461-492）
<button onClick={() => setLiveSubMode('pressure')}>压力测试</button>
<button onClick={() => setLiveSubMode('hypothesis')}>假设构建</button>
```

- **默认值**：`'pressure'`，保持向后兼容。
- **状态管理**：切换子模式时分别清除对方的激活会话 ID，防止视图冲突。
- **渲染分支**：
  - `liveSubMode === 'pressure'` → 渲染 `PressureTestWorkbench` / `PressureSessionRunner`
  - `liveSubMode === 'hypothesis'` → 渲染 `HypothesisWorkbench` / `HypothesisSessionRunner`

### 历史恢复

通过 LabContext 提供的 `continueHypothesisSession` / `viewHypothesisSession` 方法，从左侧历史面板打开会话时自动切换到 `liveSubMode === 'hypothesis'` 并加载对应会话。

## 7. 数据存储

### Bucket Key

商业假设构建器的所有会话数据存储在独立的 localStorage key 中，与压力测试完全隔离：

| 用途 | Key |
|------|-----|
| 会话数据 | `thinking-lab-hypothesis-sessions` |
| 评测日志 | `thinking-lab-hypothesis-eval-log` |

压力测试的对应 Key 分别为 `thinking-lab-pressure-engine-sessions` 和 `thinking-lab-pressure-engine-eval-log`。

### 数据结构

```typescript
// hypothesisTypes.js — 数据契约定义
interface HypothesisSession {
  id: string                     // hs_{timestamp}_{random}
  name: string                   // 截断至 28 字
  createdAt: number              // Date.now()
  updatedAt: number              // Date.now()
  originalIdea: string           // 用户原始想法
  rounds: HypothesisRound[]      // 4 轮 x 2 问
  canvas: BusinessCanvas | null  // 画布（8 问完成后填充）
  status: 'ideating' | 'questioning' | 'completed'
  promptsVersion: string         // '1.0.0-mvp'
}

interface HypothesisRound {
  roundIndex: 1..4
  roundName: string              // 用户与痛点/解决方案/商业模式/验证计划
  isCompleted: boolean
  questions: [HypothesisQuestion, HypothesisQuestion]  // 每轮恰好 2 问
}

interface HypothesisQuestion {
  questionIndex: 1..2
  questionText: string
  answerText?: string
  answeredAt?: number
}
```

### 变更通知

Store 层通过 CustomEvent `'thinking-lab-hypothesis-sessions-changed'` 通知其他组件数据已更新。

### 8.2 质检维度调整

质检函数入口位于 `hypothesisQC.js` 的 `evaluateHypothesisQuestion` 函数（lines 133-149）：

```javascript
export function evaluateHypothesisQuestion({ question, userLastAnswer, previousQuestions, roundIndex }) {
  const q = (question || '').trim()
  const failed = []

  if (q.length < QC_RULES.MIN_LENGTH || q.length > QC_RULES.MAX_LENGTH) failed.push('length')
  if (hasForbiddenWords(q)) failed.push('forbidden')
  if (!looksLikeQuestion(q)) failed.push('not_question')
  if (!hasCoBuildTone(q)) failed.push('co_build_tone')
  if (!hasUserReference(q, userLastAnswer)) failed.push('reference')
  if (isDuplicateQuestion(q, previousQuestions)) failed.push('duplicate')

  return { passed: failed.length === 0, failedChecks: failed }
}
```

**当前 6 个质检维度：**

| 维度 | 检查项 | 说明 |
|------|--------|------|
| length | 长度 | 追问须在 `[MIN_LENGTH, MAX_LENGTH]` 范围内，默认 30-120 字 |
| forbidden | 违禁词 | 禁止包含否定性/打压性用语（如「有问题」「不可能」等） |
| not_question | 问句完整性 | 须包含疑问语气（?/？/吗/什么/哪些/如何等） |
| co_build_tone | 共建语气 | 须包含「我们」「一起」「来」等共建词汇 |
| reference | 引用 | 须引用用户上文至少连续 3 个相同字符的词句 |
| duplicate | 重复度 | 不能与之前任一追问高度相似（charBigramSimilarity > 0.82 判为重复） |

**如需调整：**

1. 修改 `QC_RULES` 常量（位于 `src/config/businessHypothesisPrompts.js`，lines 258-263）中的 `MIN_LENGTH` / `MAX_LENGTH` / `FORBIDDEN_WORDS` / `CO_BUILD_KEYWORDS`
2. 或在 `evaluateHypothesisQuestion` 函数中增减检查项
3. 如需变更 Retry 提示文案，修改 `buildRetryHint` 函数（lines 155-163）

## 9. 已知限制 / 待优化点

### 9.1 画布展示：Markdown 段落解析

**现状**：`BusinessCanvasViewer.jsx` 使用自定义的 `parseSections` / `renderCanvasMdContent` 函数对 AI 返回的 Markdown 进行简单段落、列表和表格解析。该实现为轻量级手工解析，不支持嵌套 Markdown 语法、内联格式、代码块等。

**建议升级方案**：

```jsx
// 替换 BusinessCanvasViewer.jsx 中的手工解析
import ReactMarkdown from 'react-markdown'
import remarkGfm from 'remark-gfm'

export default function BusinessCanvasViewer({ session }) {
  const markdown = session?.canvas?.rawMarkdown || ''
  if (!markdown) return null

  return (
    <ReactMarkdown remarkPlugins={[remarkGfm]} className="prose prose-sm">
      {markdown}
    </ReactMarkdown>
  )
}
```

### 9.2 左侧工具列表：假设构建历史入口

**现状**：左侧工作台中间工具列中，压力测试历史入口 ID 为 `'practice'`，假设构建历史入口 ID 为 `'hypothesis-practice'`，分别由 `PressureSessionHistoryPanel` 和 `HypothesisSessionHistoryPanel` 负责渲染，通过不同图标触发。

**未来优化方向**：可考虑与压力测试共用入口，根据当前 `liveSubMode` 自动切换显示对应的历史列表。即一个「练习记录」入口，内部自动切换渲染面板。如需实现，修改文件为 `src/components/workbench/workbenchToolBodies.jsx` 中的 `WorkbenchToolPaneContent` 函数。

### 9.3 状态机制差异

**现状**：假设构建器初始状态为 `'ideating'`，首问生成后才转为 `'questioning'`。这与压力测试的 `'deconstructing'` → `'questioning'` 不同。

| 状态 | 假设构建器 | 压力测试 |
|------|-----------|---------|
| 初始状态 | `ideating` | `deconstructing` |
| 追问中 | `questioning` | `questioning` |
| 完成 | `completed` | `completed` |

**注意**：`HypothesisSessionRunner.jsx` 在 `useEffect` 中检测 `session.status === 'ideating'` 时自动触发 `bootstrapHypothesisSession`（lines 52-81）。如果未来需要保留初始页面让用户手动点击开始，需修改此逻辑。

### 9.4 其他待优化点

- **Layer 2 抽检**：`maybeAppendLayer2SampleRecord` 目前为占位实现（`stub_no_model`），未接入真正的评测模型；可替换为队列/离线打分系统。
